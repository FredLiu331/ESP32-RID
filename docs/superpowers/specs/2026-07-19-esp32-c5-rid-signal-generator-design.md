# 基于 ESP32-C5 的 RID 信号发生器设计

## 1. 项目目标与范围

本项目用于实现一款远程识别（RID）信号发生器，测试用户自研的 RID 接收机。设备能够模拟 1～50 架活跃无人机，并通过可配置的协议与无线承载组合广播合成飞行数据。

设备仅用于经过授权且边界受控的室外试验场，不用于无人值守地运行在不受控公共区域。测试身份不得与真实无人机身份重复。

首版采用以下基线：

- 微雪 ESP32-C5-WIFI6-KIT-N32R8 开发板
- ESP-IDF v5.5.4
- C/C++
- USB 5 V 供电
- 0～50°C 工作环境
- 串口交互式 Shell 作为唯一配置入口

AT 命令、终端用户界面（TUI）和网页配置不纳入首版范围。后续如需增加，可复用同一配置服务。

## 2. 功能需求

设备应满足以下要求：

1. 可配置模拟 1～50 架活跃无人机。
2. 支持 GB 46750-2025 和 ASTM F3411-22a 定义的 OpenDroneID 消息格式。
3. 每架模拟无人机只选择一种 RID 协议。
4. 每架模拟无人机只选择一种无线承载：
   - Bluetooth 4 Legacy Advertising
   - Bluetooth 5 Extended Advertising
   - 2.4 GHz Wi-Fi，固定信道 6
   - 5 GHz Wi-Fi，固定信道 149（5745 MHz）
5. OpenDroneID 的 Wi-Fi 承载同时支持 Beacon 和 NAN，默认使用 Beacon。
6. 使用内置轨迹模型生成合成且运动关系合理的飞行数据。
7. 使用配置组批量创建无人机，不要求逐架配置。
8. 将最后一次提交成功的配置持久化到 NVS。
9. 上电后自动恢复并启动最后一份有效配置。
10. 启动后持续发射，不设置会话超时。
11. 使用固定低发射功率，标称有效测试距离不超过 10 m。
12. 提供实时状态和汇总发送统计，但不保存历史会话日志。

## 3. 约束与非目标

- 单颗 ESP32-C5 无法同时驻留在 Wi-Fi 信道 6 和 149。固件通过快速切换信道实现逻辑并行。
- Wi-Fi 与 BLE 通过 ESP-IDF 共存机制工作，物理层可能采用时分调度。
- 相对时间测试模式不适用于要求准确 UTC 时间的协议一致性认证。
- 自研 RID 接收机是唯一外部业务验收基准，第三方接收机和抓包工具不属于交付依赖。
- 不支持导入真实无人机序列号。
- 运行实例可以查看，但首版不支持逐架直接编辑。
- 专用硬件改版、外壳、电池供电和工业温度认证不在首版范围内。

## 4. 总体架构

系统采用分层数据流水线：

```text
串口 Shell / NVS
       |
       v
配置组
       |
       v
机群展开器（生成 1～50 架运行实例）
       |
       v
轨迹引擎
       |
       v
统一 RID 飞行状态
       |
       +-------------------+
       v                   v
GB 46750 编码器     OpenDroneID 编码器
       |                   |
       +---------+---------+
                 v
          截止时间调度器
                 |
       +---------+---------+
       v                   v
 BLE4 / BLE5       Wi-Fi Beacon / NAN
       |                   |
       +---------+---------+
                 v
             射频协调器
```

### 4.1 配置服务

配置服务负责持久化结构、校验、暂存、应用、保存和回滚。其他模块只读取不可变配置快照，不直接访问 NVS。

### 4.2 机群展开器

机群展开器把配置组转换为运行实例。每个运行实例包含派生测试 ID、所选协议、所选承载、轨迹状态、启用的消息类型、调度状态和统计计数。

### 4.3 轨迹引擎

轨迹引擎只生成与协议无关的位置和运动数据，不依赖编码器或射频实现。

### 4.4 协议编码器

GB 46750 与 OpenDroneID 使用相互独立的模块，并实现统一编码接口。编码器接收统一飞行状态，输出一个或多个不可变报文载荷。

OpenDroneID 编码应复用官方 `opendroneid-core-c`。引入前须在依赖声明中记录其许可证，以 Git 提交哈希锁定具体版本，并使用 ASTM F3411-22a 黄金向量验证。具体无线封装不属于核心编码器职责。

### 4.5 截止时间调度器

调度器维护每条已启用消息流的下次发送截止时间，将编码结果路由至对应承载队列，并统计应发、实发、延迟、失败和丢弃次数。

### 4.6 射频协调器

射频协调器负责 Wi-Fi 信道切换、BLE/Wi-Fi 共存、承载生命周期及故障恢复。协议编码器不得直接调用 ESP-IDF 射频 API。

## 5. 配置模型

配置分为全局配置、配置组和运行实例三层。

### 5.1 全局配置

- 试验场中心经纬度
- 全局默认消息周期
- 信道 6 与信道 149 的驻留策略
- 固定板级射频功率配置
- 自动启动状态

基线固定功率为 Wi-Fi 传导功率 2 dBm、BLE 传导功率 0 dBm。板级测试应确认 10 m 内可靠接收。如需修正，只能修改固件构建配置，不通过 Shell 动态调整。

### 5.2 配置组

每个具名配置组包含：

- 无人机数量
- 协议：GB 46750 或 OpenDroneID
- 承载：BLE4、BLE5、Wi-Fi 2.4 GHz 或 Wi-Fi 5 GHz
- 适用时的 Wi-Fi 封装方式：Beacon 或 NAN
- 轨迹模型及参数
- 启用的消息类型
- 组级默认消息周期
- 单消息周期覆盖值

所有配置组的数量总和必须为 1～50，才能启动配置。

### 5.3 运行实例

运行实例由当前配置快照自动生成。测试 ID 根据设备身份、组名和组内索引确定性派生。因此，配置不变时，重启后生成的 ID 保持一致。

用户可以查看单架无人机的 ID、当前状态、发送截止时间和统计信息。首版不直接编辑运行实例；修改配置组后，系统一次性重新生成整个机群。

## 6. Shell 设计

Shell 将待编辑配置与当前运行配置分离。`config apply` 完成校验后原子替换运行机群，`config save` 将已经应用的快照提交到 NVS。

命令形式示例：

```text
group add odid_ble count 10 protocol odid transport ble5
group set odid_ble trajectory circle radius 30 speed 5 altitude 80
group set odid_ble rate location 500ms

group add gb_wifi count 5 protocol gb46750 transport wifi5
group set gb_wifi wifi-mode beacon

group list
group delete gb_wifi
config check
config apply
config save
config rollback

fleet list
fleet show 7
radio status
stats show
system stop
system start
```

以下情况必须拒绝配置：

- 无人机总数为 0 或超过 50
- 配置组重名
- 坐标、速度、高度、周期或轨迹参数无效
- 协议、消息、承载或封装方式组合不受支持
- 未设置试验场中心坐标
- 估算负载超过实现的硬限制

如果负载未超过硬限制但可能造成延迟或丢包，交互式 Shell 应给出警告并要求确认。压力测试模式允许用户确认后继续。

## 7. 合成飞行数据

用户通过 Shell 设置试验场中心坐标。轨迹引擎在局部东-北-天坐标系中计算运动，再转换为地理坐标。

首版包含以下轨迹：

- 悬停
- 直线往返
- 圆周
- 矩形航线
- 航点循环
- 起飞、巡航和降落循环

同组实例使用确定性的起点偏移、相位偏移和启动延迟，避免所有无人机状态完全重叠。轨迹参数按模型包含原点偏移、高度、速度、航向、尺寸或半径、航点列表和启动延迟。

统一飞行状态至少包含：

- 经度和纬度
- 海拔高度和相对起飞点高度
- 水平速度
- 垂直速度
- 航向
- 空中或地面状态
- 协议要求时的操作者或系统位置
- 相对消息时间

本设计不包含可掉电保持的绝对时间源。每次启动后，时间从配置的测试时刻或相对零点重新开始，并在 Shell 状态中显示为未同步。这是有意设计的接收机压力测试模式。

## 8. 协议支持

### 8.1 OpenDroneID

编码器支持全部标准消息类型：

- Basic ID
- Location
- Authentication
- Self-ID
- System
- Operator ID

每种消息均可按配置组启用或禁用。消息周期默认继承全局配置，并允许通过组级默认值和单消息设置覆盖。

Wi-Fi 同时支持 Beacon 和 NAN；配置组未指定时使用 Beacon。BLE4 使用 Legacy Advertising，BLE5 使用 Extended Advertising。每架运行实例只使用一种承载。

### 8.2 GB 46750-2025

GB 编码器实现 GB 46750-2025 规定的完整广播消息集合和字段约束。它与 OpenDroneID 共用统一飞行状态，但保留独立编码器和测试套件。

配置校验器必须执行标准规定的协议、承载和字段兼容规则，不得为了接受配置而生成非标准组合。

## 9. 调度与射频运行

### 9.1 消息调度

消息周期可用于压力测试。首版要求支持 100 ms～60 s 的周期范围，实际分辨率和边界由 `config check` 显示。

周期配置的优先级从高到低为：

1. 单消息覆盖值
2. 配置组默认值
3. 全局默认值

每个承载队列内部使用最早截止时间优先调度。轨迹计算不受发送拥塞阻塞。

### 9.2 BLE 复用

BLE 运行实例复用数量受控的控制器广播实例。系统按消息截止时间轮换载荷，不为每架无人机永久占用一个独立广播集。

### 9.3 Wi-Fi 信道切换

信道 6 与信道 149 的队列共享一套 ESP32-C5 Wi-Fi 射频。每个活跃信道的初始驻留目标为 100 ms，射频协调器根据待发送载荷数量和最近截止时间调整驻留比例。如果只有一个 Wi-Fi 频段存在待发实例，则保持驻留在该频段。

信道切换必然产生接收空窗，因此本系统只声明逻辑双频并行，不声明物理双频并发。`radio status` 和 `stats show` 应显示驻留时间、切换耗时、实发数量和丢弃数量。

信道 149 使用 ESP-IDF 的 `CN` 国家区域设置。固件不得绕过国家区域的信道或功率限制。

### 9.4 过载策略

当队列超过固定容量时，调度器优先丢弃最过期的载荷，从而保留较新的飞行状态，避免继续发送陈旧数据。每次事件都增加明确的延迟和丢弃计数。

## 10. 持久化与启动

NVS 保存配置结构版本、配置载荷、完整性校验值和提交代次。保存操作应原子替换上一份有效快照。

启动行为如下：

- 存在有效配置时，自动展开机群并开始发射。
- 新设备没有已保存配置时，只启动 Shell，保持停发。
- 配置损坏或版本不兼容时，禁止发射并在 Shell 中报告原因。
- 在线应用新配置失败时，继续运行上一份有效配置。

设备不保存历史会话日志。配置与统计使用不同生命周期；统计在重启或显式复位命令后清零。

## 11. 故障处理

- 单条消息编码失败时，只跳过该载荷并增加编码错误计数。
- 射频提交暂时失败时，仅在载荷仍然有效的期间重试。
- Wi-Fi 信道切换或 BLE 广播连续失败时，重启对应无线子系统。
- 在规定重试次数内无法恢复时，触发任务看门狗并受控重启设备。
- 无效运行状态不得产生未经校验的载荷；编码器返回错误，由调度器统计。
- Shell 命令返回稳定的错误码和简洁诊断信息。

## 12. 可行性验证关卡

在实现完整系统前，必须首先使用微雪开发板和 ESP-IDF v5.5.4，通过公开 ESP-IDF API 验证：

1. 在信道 6 发送原始 Beacon。
2. 在信道 149 发送原始 Beacon。
3. 在标准和国家区域规则允许的频段发送兼容 NAN 的 Action 帧。
4. 使用 BLE Legacy Advertising 发送有效 RID 载荷。
5. 使用 BLE Extended Advertising 发送有效 RID 载荷。
6. Wi-Fi 在信道 6 与 149 间切换时，BLE 广播仍可持续工作。

ESP-IDF 文档说明 `esp_wifi_80211_tx()` 支持 Beacon 和 Action 帧，但完整 RID 封装及共存行为仍须在指定开发板和 IDF 版本上实测。不得把私有驱动符号或二进制补丁作为备选方案。

任一验证关卡失败时，应记录已验证的限制并修改支持矩阵，再继续实现机群功能。

## 13. 验证与验收

### 13.1 主机单元测试

- 配置结构和校验边界
- 测试 ID 的确定性与唯一性
- 所有轨迹模型和坐标转换
- 时间回绕及未同步时间行为
- OpenDroneID 黄金向量编码
- GB 46750 黄金向量编码
- 周期覆盖优先级、截止时间排序和过载策略

### 13.2 板上集成测试

- 纯 BLE4 和纯 BLE5 机群
- 信道 6 的 Beacon 和 NAN 机群
- 信道 149 的 Beacon 和 NAN 机群
- BLE 与单一 Wi-Fi 频段混合机群
- BLE 加信道 6/149 跳频的混合机群
- 配置应用、回滚、保存、掉电和自动恢复

### 13.3 系统验收

分别运行具有代表性的 1 架、10 架和 50 架场景。自研 RID 接收机应满足：

- 在基准验收配置下发现全部已配置无人机。
- 每条观测均能关联到预期的确定性测试 ID。
- 正确解析已启用消息及配置字段。
- 观测到的轨迹不存在非法坐标或状态跳变。
- 消息间隔与发生器统计及跳频空窗一致。

基准验收配置采用：主要动态消息周期 1 s、其他启用消息周期 3 s、Wi-Fi 默认驻留目标 100 ms、无明显遮挡测试距离 10 m。50 架场景应在 15 s 内发现全部 ID，10 分钟内不得产生格式错误的已解码载荷，主要动态消息送达率不低于应发数量的 95%。任意压力测试周期只按发生器统计进行特性记录，不要求达到该基准送达率。

发生器自身应满足：

- 连续运行 24 小时，无未处理复位或资源泄漏。
- 连续执行 10 次掉电重启后，每次均恢复已保存机群并自动开始发射。
- 按承载报告应发、实发、延迟、丢弃、编码错误和射频错误计数。
- 在 USB 5 V 供电、0～50°C 环境下工作。
- 固定功率校准后，在授权试验场 10 m 距离可靠接收。
